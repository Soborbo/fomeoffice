# Code audit — 2026-06

Scope: full pass over `src/`, `migrations/`, `wrangler*.json`, the client ↔
API contracts on every page (including the new walk-in image picker / price
ladder / walk-in log on `/board`), auth/RBAC, cron, and the third-party
integrations (Resend, Stripe, Google Sheets, Anthropic OCR, Xero export).
Follow-up to `docs/AUDIT-2025-05.md`; all items fixed there were re-verified
and none regressed.

## High — fixed in this change

### H1. Dashboard cash reconciliation disagrees with the daily form

**File**: `src/lib/db/cash-deposits.ts:193`.

`getCashFlowSummary` counted "cash collected" with
`payment_method = 'cash'`, which excludes legacy `done` bookings whose
`payment_method` is NULL. The daily form's `computeExpectedTotals`
(`src/lib/db/daily.ts:37`) deliberately counts those rows as cash (the
2025-05 audit's C1 decision). The comment in `cash-deposits.ts` even
claimed the two definitions matched — they didn't. Result: the dashboard
reconciliation card undercounts collected cash and shows phantom variance
against `expected_cash` whenever legacy rows exist in the range.

**Fix**: the bookings subquery now uses
`(payment_method = 'cash' OR payment_method IS NULL)`, matching `daily.ts`
(bookings only ever hold `cash`, `card`, or NULL — enforced by the
`/api/bookings` schema). Comment corrected.

### H2. CSV formula injection in the Xero export

**File**: `src/lib/xero/csv-export.ts` (`csvEscape`).

`csvEscape` handled quoting but not formula injection. A vendor name,
expense description, or customer-derived field beginning with `=`, `+`,
`-`, `@`, tab, or CR is executed as a formula when the exported CSV is
opened in Excel/Sheets — a classic data-exfiltration / code-execution
vector against the accountant. These fields are user-controlled (e.g.
walk-in customer email, expense vendor typed by staff).

**Fix**: cells matching `/^[=+\-@\t\r]/` are prefixed with a single quote
unless they are plain numbers — `penceToPounds` legitimately emits
negative amounts like `-5.00`, which must stay numeric for accounting.

### H3. Scheduled emails never fire — cron was never wired up

**Files**: `wrangler.json` (empty `triggers.crons`), new
`workers/cron-pinger.mjs`, new `wrangler-cron.json`, `package.json`.

The code ships three cron tasks (`close-reminder`, `super-escalation`,
`daily-summary`) behind the token-authenticated `/api/cron/run` endpoint,
and the plan (`references/CRM-PATTERNS.md §10`) calls for Cloudflare cron
triggers — but nothing ever invokes the endpoint. `wrangler.json` has
`"crons": []`, and the main worker (Astro's `entry.mjs`) exports no
`scheduled` handler, so adding triggers there alone would not work either.
Net effect: no close reminders, no super-admin escalation, no daily
summary email — ever — unless someone manually curls the endpoint.

**Fix**: added a minimal companion worker, `workers/cron-pinger.mjs`,
deployed from `wrangler-cron.json` (`npm run deploy:cron`). It owns three
UTC cron triggers and POSTs the matching task to
`${SITE_URL}/api/cron/run` with the `x-cron-secret` header:

| Cron (UTC)    | Task               | Local effect (shop closes 19:00 UK)      |
|---------------|--------------------|------------------------------------------|
| `5 19 * * *`  | `close-reminder`   | 19:05 GMT / 20:05 BST — after close both ways |
| `20 19 * * *` | `super-escalation` | 19:20 GMT / 20:20 BST                     |
| `0 22 * * *`  | `daily-summary`    | 22:00 GMT / 23:00 BST — still same day    |

Both reminder tasks no-op when the daily form is already filled, so the
one-hour BST drift is harmless. Deployment requires setting the same
`CRON_SECRET` on both workers (documented in `.env.example`).

## Medium — fixed in this change

### M1. Close-reminder email links to the wrong domain

**File**: `src/lib/cron/close-reminder.ts:77`.

The reminder email hardcoded
`https://carwash.golaxo.workers.dev/app/daily` while the production site
is `https://bristolcarwash.co.uk` (`SITE_URL` in `wrangler.json`). The
workers.dev link breaks the moment the workers.dev route is disabled, and
sends staff to the wrong origin (separate cookie jar → forced re-login).

**Fix**: the link now uses `SITE_URL` from the worker env, falling back to
the production domain.

### M2. `.env.example` missing four documented secrets

**File**: `.env.example`.

`CRON_SECRET`, `ANTHROPIC_API_KEY`, `ADMIN_EMAIL`, and `ADMIN_PASSWORD`
are declared in `src/env.d.ts` and used by `/api/cron/run`,
`/api/app/ocr/receipt`, and `/api/auth/login`, but were absent from
`.env.example`. A fresh deployment following the example file gets a
503 from cron and OCR with no hint why.

**Fix**: all four documented with setup notes.

## Findings flagged but not changed

### F1. Unused `IMAGES` and `SESSION` bindings

**File**: `wrangler.json:31-36`.

The `SESSION` KV namespace and the `IMAGES` (Cloudflare Images) binding
are declared but never read — sessions live in D1 and image compression
is client-side (`src/lib/image/compress.ts`). Harmless clutter; left in
place since removing bindings is a deploy-side decision, but neither is
load-bearing.

### F2. No rate limiting on `/api/staff/login`

PIN login runs PBKDF2 per attempt with no IP throttling. Brute-forcing a
4-digit PIN is feasible over hours. Mitigations already in play: PBKDF2
cost, worker CPU budget, and the 2025-05 recommendation to move to
per-worker PINs. Recommended next step: a KV-backed counter (the unused
`SESSION` namespace would do) limiting attempts per IP, or a 6+ digit PIN
policy.

### F3. Module-level Google Sheets token cache

**File**: `src/lib/google-sheets.ts`.

The 50-minute access-token cache lives in a module-level variable. On
Workers this is per-isolate and dies with the isolate — standard practice,
not a leak. No change; noted only because cache lifetime is bounded by
isolate recycling, not by the 50-minute TTL.

### F4. OCR model fallback is pinned

**File**: `src/pages/api/app/ocr/receipt.ts:65`.

Model comes from the `ocr_model` setting with a pinned fallback
(`claude-haiku-4-5-20251001`). Fine today; when the model is eventually
deprecated, update the setting (no deploy needed) or the fallback.

### F5. `wrangler.json` retains an empty `triggers.crons`

Left as-is deliberately: the main Astro worker has no `scheduled` export,
so triggers must never be added there — they live on the cron-pinger
worker. The empty array documents that.
