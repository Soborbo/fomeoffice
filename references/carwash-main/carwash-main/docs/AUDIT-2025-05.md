# Code audit — 2025-05

Scope: full pass over `src/`, `migrations/`, route handlers, auth, RBAC,
and the booking → board → CRM data flow. Triggered by the report that
website orders arrive but never appear in the login-protected CRM.

## Critical — fixed in this change

### C1. Bookings marked "Done" on `/board` are invisible to the CRM

**Files**: `src/pages/api/bookings.ts`, `src/pages/board.astro`,
`src/lib/db/bookings.ts`, `src/lib/db/daily.ts`, `src/lib/db/dashboard.ts`.

The `/board` PIN flow exposes only **Done** and **No Show** buttons. The
"Done" path posts `{action: 'setStatus', status: 'Done'}` to
`/api/bookings`, which updates `bookings.status='done'` and leaves
`payment_method=NULL`. Every CRM revenue query filters by
`status='done' AND payment_method IS NOT NULL`, so the row never counts
toward expected cash, the admin dashboard KPIs, or any chart. The bookings
*are* in D1 — they're just filtered out everywhere the user looks.

**Fix**:

1. `/board` now opens a Cash/Card modal when "Done" is tapped (mirrors the
   walk-in flow). No Show and Undo bypass the modal.
2. `/api/bookings` `setStatus` accepts an optional `payment_method`. When
   transitioning to `done` it **requires** the field (returns
   `PAYMENT_METHOD_REQUIRED`); when transitioning to any other status it
   clears `payment_method` so Undo doesn't leave stale data.
3. `updateBookingStatus` now writes both `status` and `payment_method` in
   one statement and refuses to touch soft-deleted rows.
4. Revenue queries in `daily.ts` and `dashboard.ts` drop the
   `payment_method IS NOT NULL` filter. Legacy `done` rows with NULL
   payment method count as **cash** (conservative default — most forecourt
   transactions are cash). New bookings always have an explicit value.
5. `xero/csv-export.ts` keeps the strict filter — accounting wants explicit
   data, not inferred defaults.

### C2. HTML injection in `/api/booking.ts` and `/api/contact.ts` emails

**Files**: `src/pages/api/booking.ts`, `src/pages/api/contact.ts`,
`src/lib/utils/html.ts` (new).

Both endpoints interpolate user-controlled fields (first name, last name,
phone, email, vehicle label, service label, free-text message) directly
into HTML email bodies via template literals — no escaping. The
office staff received whatever HTML the booker wrote, including
`<script>` and event handlers. Most modern webmail clients sanitize
inbound HTML, but Outlook desktop and many self-hosted clients do not,
so this was a real XSS vector against staff.

Subject lines were also assembled from raw user input (`subject: \`New
Booking: ${data.name} - …\``). A name containing `\r\n` could inject
arbitrary email headers (Bcc, Reply-To rewrite, etc.).

**Fix**:

1. New `src/lib/utils/html.ts` with `escapeHtml` (5-char HTML escape)
   and `safeHeader` (collapses CR/LF to spaces for use in subject
   lines and addresses).
2. Every user-controlled string in both email HTML bodies now goes
   through `escapeHtml`. The walkin invoice template
   (`src/lib/email/invoice.ts`) and damage email
   (`src/lib/email/damage.ts`) already escape — only the two public
   endpoints were missing it.
3. All subject lines wrap user inputs with `safeHeader`.

### C3. Booking endpoint accepts past dates

**File**: `src/pages/api/booking.ts`.

The Zod schema validated date format but not range, so a direct POST
could create a booking for any past day. The website form gates this
client-side but a script bypassing the form (or an attacker) could
backdate bookings, which then show up on `/board` and corrupt daily
totals.

**Fix**: rejected when `date < today` (Europe/London timezone, the same
ops timezone used everywhere else).

## High — fixed in this change

### H1. Race on `customers(email)` unique index

**Files**: `src/lib/db/bookings.ts:144`, `src/lib/db/walkins.ts:97`.

`findOrCreateCustomerByEmail` does `SELECT … LIMIT 1` then `INSERT`. Two
concurrent bookings for a brand-new email both miss the lookup, both try
to insert, and one throws on the unique index. The booking handler had no
catch for it.

**Fix**: switched both helpers to `INSERT OR IGNORE` followed by a
re-select. Either insert wins or we read the winner's id.

### H2. `updateBookingStatus` can mutate soft-deleted rows

**File**: `src/lib/db/bookings.ts:245`.

The function did `UPDATE bookings SET status = ? WHERE id = ?`, ignoring
`deleted_at`. A soft-deleted booking can't be re-surfaced from the board
(it's not in `live_bookings`), so the practical risk was low, but a
direct id call from elsewhere could still mutate it.

**Fix**: added `AND deleted_at IS NULL` to the WHERE clause.

## Low — fixed in this change

### L1. `/board` date filter uses `YYYY-M-D` keys

**File**: `src/pages/board.astro`.

`toDateKey` and `getDateKey` produced `YYYY-M-D` (no zero-padding and
zero-indexed month). Comparisons were symmetric so filtering worked, but
inconsistent with the rest of the codebase.

**Fix**: both helpers now produce canonical `YYYY-MM-DD`.

## Findings flagged but not changed

### M1. PIN verification is O(N) PBKDF2

**File**: `src/lib/auth/worker-pin.ts`.

`verifyWorkerPin` scans every active worker and runs PBKDF2 (100k
iterations) per row. For 15 workers that's roughly 1–2 s of CPU per
request, on a worker that has a CPU budget. The file already has a TODO
acknowledging this. Acceptable for a single-shop deployment but worth
adding a lookup index (e.g. a short truncated-PIN prefix hash) before
opening this surface to abuse.

### M2. `/api/staff/login` legacy `BOARD_PIN` maps to the first active worker

**File**: `src/pages/api/staff/login.ts:104`.

When the legacy single-PIN fallback is enabled, the session is attached
to the lowest-id active worker. Audit logs will misattribute actions.
Already documented in the guide's troubleshooting table; switching to
per-worker PINs and clearing `BOARD_PIN` is the recommended path.

### L2. `/board` auto-refresh runs every 5 minutes

**File**: `src/pages/board.astro:595`.

That's slow for a busy shop — a worker might mark a walk-in then wait 5
minutes to see new bookings. A 30–60 s polling interval is more typical.
Pure UX call; not changed because there's no failure mode.

### L3. `/board` notification sound is loud

**File**: `src/pages/board.astro:546-558`.

Two beeps at 520 Hz / 660 Hz with full gain on every new booking. Easy to
disable per-tablet if it turns into a nuisance.

### L4. Booking confirmation email wording

**File**: `src/pages/api/booking.ts:89`.

Says "Your appointment is confirmed" but the shop doesn't have time-slot
booking — customers just turn up. Wording is misleading but not broken.
